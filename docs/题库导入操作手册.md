# 题库导入操作手册

本文档用于说明向知简题库新增或更新题目时的统一规则。适用于言语理解、数量关系、资料分析、逻辑判断，以及以后新增的其他板块。

## 一、导入总原则

每道题必须作为一个完整题目块处理，并将同一题的全部内容绑定在一起：

- 题干
- A-D（或实际存在的全部）选项
- 正确答案
- 题型分类
- 普通解析
- 实战解析
- 花生批注
- 题干中的下划线、粗体、括号和重点标注
- 题目需要的图表、图片或公式资源

以完整解析 Word 为主要来源。题本用于核对题目、选项和题数；PDF 原稿主要用于视觉复核和图片裁切，不要求重新解析 PDF。

导入完成后，必须同时检查题库数据、历史记录、错题本、图片资源和线上接口，确认旧题 ID 仍然有效，再提交 GitHub、更新 D1 并部署 Worker。

## 二、发布链路

标准顺序如下：

来源确认 → 题块切分 → 题目与答案绑定 → 解析绑定 → 花生批注匹配 → 格式复核 → 图片裁切与 R2 上传 → 本地校验 → GitHub → 远程 D1 → Cloudflare Worker 部署 → 线上冒烟测试

遇到错误时，回到对应环节修复并重新执行后续校验；未通过本地校验时不得推送或部署。

## 四、文件准备规则

### 4.1 来源优先级

1. 完整解析 Word：作为题干、选项、答案、解析、批注和格式的主要来源。
2. 题本 Word：用于核对题目数量、题号、题干和选项。
3. PDF 原稿：用于核对页面视觉效果、裁切图表和确认 Word 转换是否丢失内容。

优先使用 Word 进行结构化导入。PDF 中的内容如果已经可靠地转成 Word，不必再次从 PDF 解析文字。

### 4.2 一个文件包含多道题

不能假设一个 Word 文件只包含一道题。必须按题号、答案锚点、解析锚点和题型锚点切分出多个完整题块，再逐题导入。

导入前确认：

- 题号是否连续或能建立明确映射；
- 每题选项是否完整；
- 答案是否存在且只属于当前题；
- 解析是否从当前题开始、在下一题开始前结束；
- 花生批注是否能定位到当前题；
- 图表或图片是否属于当前题。

页眉、页脚、页码、宣传语、浮动文本框、左栏批注和跨页重复文字不能当作题干或解析导入。

## 五、每题数据结构

每道题至少应形成以下字段：

| 字段 | 内容 |
| --- | --- |
| stem | 纯文本题干 |
| options | 选项数组，保留选项顺序 |
| answer | 正确选项，通常为 A-D，也允许实际题目使用的 A-H |
| type | 题型分类 |
| explanation | 普通解析，不能为空 |
| details.practicalAnalysis | 实战解析，存在时完整保留 |
| details.notes | 花生批注数组或结构化批注 |
| stemRich | 带格式的题干内容 |
| annotatedStem | 解析页面需要展示的标注题干 |
| imageKey | 题目关联的 R2 图片 key；无图时为空 |

解析中的题目、下划线、括号内问题和高亮内容，应存入前端可以安全渲染的富文本字段，不能只保留去格式后的纯文本。

## 六、题块切分与绑定

### 6.1 题目切分

逻辑判断、数量关系等通常优先使用 【题型分类】、题号和答案等稳定锚点。资料分析通常使用材料组、题号和 【参考答案及正确率】 等答案锚点。

题号缺失或 OCR/Word 转换导致题号不可读时，使用当前题块的前后边界、选项结构、答案锚点和顺序索引共同定位。不得仅凭页面位置盲目拼接。

### 6.2 答案和解析绑定

答案只绑定到当前题块。解析必须从当前题的解析标题或内容开始，到下一题块开始前结束。

如果题本和解析的题数不一致，先建立逐题映射并列出缺失项、重复项和无法确认项；不能把上一题解析顺延给下一题，也不能用空解析填充后直接上线。

### 6.3 花生批注匹配

花生批注使用“序号定位 + 内容相关度复核”的混合规则：

1. 先读取批注前面的题号、序号或题块标记，尝试定位所属题目。
2. 将批注内容与候选题的题干、选项、普通解析和实战解析进行相关度复核。
3. 序号和内容一致时直接绑定。
4. 序号与内容冲突时，以内容相关度为准，并记录冲突。
5. 序号无法判断时，使用题干与批注内容的关键词、论证关系、数字、专名和语义相似度匹配。
6. 仍然不确定时不强行归题，进入“未绑定批注”清单，等待人工复核。

批注处理时必须去除页脚、页码、下一题选项、重复批注和与当前题无关的浮动文本。批注不能因为出现在同一页就自动归属于该页第一题或最后一题。

## 七、格式与解析展示规则

必须尽量保留 Word 中影响做题和解析理解的格式：

- 下划线；
- 粗体；
- 括号及括号内的文字；
- 论据、结论、归因、前提、假设、转折和关键数字等重点内容；
- 解析中新出现的括号以及括号内的问题。

逻辑判断的解析展示规则应与已有言语题规则保持一致。用户点击“查看解析”后，题目中的下划线、重点高亮、批注标记和括号内问题应正确显示在题干相应位置，不能把这些内容平铺到题目末尾，也不能污染正式题干。

建议将“原始文本”“富文本题干”“解析标注题干”分开保存，避免为显示解析而破坏用户实际作答时的题干。

## 八、图表、图形和 R2 资源

### 8.1 需要上传图片的情况

以下内容不能可靠转写为普通文字时，必须提取为图片并关联到对应题目：

- 逻辑判断的矩阵、卡片、真假关系图和图形题；
- 数量关系的几何图、路线图、概率图和复杂公式；
- 资料分析的表格、饼图、折线图、柱状图、示意图和选项图形；
- Word 转换后明显缺失、错位或无法辨认的内容。

能准确转成文字的内容可以保留文字，但资料表格和图表通常建议同时保留图片，以保证用户做题时看到原始材料。

### 8.2 裁切要求

图片应只包含材料或题目所需的图表，不得包含：

- 正确答案；
- 正确率；
- 普通解析；
- 实战解析；
- 花生批注；
- 其他题目的文字。

裁切时必须检查右边框、最后一列、最后一个选项、图例、坐标轴、单位和图下注释是否完整。跨页图表需要合并，并清理页眉、页脚和页码。

### 8.3 R2 key

建议使用可读、稳定且不重复的 key，例如：

- numeric/20260717/quantity/500004.png
- numeric/20260717/data/600094.png
- numeric/20260717/logic/400092.png

同一题的所有图片都要明确关联到该题，不能只按上传顺序或页面编号关联。上传后检查 R2 返回状态为 HTTP 200，并通过项目的媒体接口访问验证。

## 九、分类规则

分类名称要稳定，新增题库应沿用已有分类，不要因为同义词重复创建分类。

数量关系可按以下方向分类：

- 几何；
- 概率；
- 行程；
- 工程；
- 排列组合；
- 容斥；
- 最值；
- 数字推理；
- 其他数量关系题型。

资料分析可按以下方向分类：

- 基期；
- 增长量；
- 增长率；
- 比重；
- 平均数；
- 倍数；
- 综合分析；
- 其他资料分析题型。

逻辑判断可按以下方向分类：

- 支持；
- 削弱；
- 归因；
- 推出；
- 前提假设；
- 解释说明；
- 分析；
- 图形题；
- 其他逻辑判断题型。

导入后检查分类统计，确保题目数量、分类数量和前端筛选结果一致。

## 十、ID、历史记录和错题本

历史记录、做题记录和错题本通过题目 ID 关联。更新题干、选项、解析、批注或图片时，必须保留原题的 question.id。

禁止以下操作：

- 删除旧题后重新生成新 ID；
- 因为分类改变而更换题目 ID；
- 只更新题库 JSON、不更新 D1；
- 只更新题目、不检查错题本和历史记录；
- 让相同题目同时存在多个稳定 ID。

更新后至少抽查：

1. 一道有历史做题记录的旧题；
2. 一道旧错题；
3. 一道有花生批注的题；
4. 一道有图片的题；
5. 一道逻辑图形题或资料图表题。

## 十一、导入前校验清单

### 11.1 数量和结构

- 数量关系应为 600 题；
- 资料分析应为 600 题；
- 其他板块按本次来源的实际题数核对；
- ID 无重复、无越界、无意外变化；
- 每题至少有 4 个选项，选项顺序正确；
- answer 是有效选项；
- 题干和选项不为空；
- 解析和实战解析不为空；
- 题型分类有效。

### 11.2 内容绑定

- 解析没有串到相邻题；
- 答案没有串题；
- 花生批注与题干和解析相关；
- 低置信度批注已进入未绑定清单；
- 下划线、粗体、括号和高亮位置可正常显示；
- 图片与题目一一对应。

### 11.3 资源和数据库

- 每个有图题目都有 imageKey；
- R2 图片访问返回 HTTP 200；
- 本地 D1 迁移执行成功；
- 本地 D1 中题数、分类数和答案数正确；
- 历史记录和错题本仍能通过原 ID 查询；
- 线上 health、ready、categories、questions、by-ids 和媒体接口正常。

## 十二、推荐执行命令

以下命令在仓库根目录执行。命令中的查询条件应根据本次导入内容调整。

~~~powershell
pnpm build

pnpm exec wrangler d1 migrations apply zhijian-db --local
pnpm exec wrangler d1 execute zhijian-db --local --command "SELECT COUNT(*) FROM questions"
pnpm exec wrangler d1 execute zhijian-db --local --command "SELECT category, COUNT(*) FROM questions GROUP BY category"

pnpm exec wrangler d1 migrations list zhijian-db --remote
pnpm exec wrangler d1 migrations apply zhijian-db --remote

git add migrations scripts src worker docs
git commit -m "Import new question bank"
git push origin main

pnpm deploy
~~~

如果 GitHub 直连失败，而本机 Clash Verge/Mihomo 代理端口为 7897，可以使用：

~~~powershell
git -c http.proxy=http://127.0.0.1:7897 push origin main
~~~

推送和部署后不要只看命令退出码，还要实际请求线上接口，检查题目、分类、解析和图片是否能在前端正常读取。

## 十三、推荐的新增题库流程

1. 收集完整解析 Word、题本 Word 和必要的 PDF 原稿。
2. 确定板块、题目总数、分类和新 ID 起始范围。
3. 按完整题块切分 Word，不把一个文件当成一道题。
4. 绑定题干、选项、答案、普通解析和实战解析。
5. 按“序号定位 + 内容相关度”匹配花生批注。
6. 复核下划线、粗体、括号、高亮和解析展示字段。
7. 提取图表、图形和无法可靠转写的内容，上传 R2 并写入 imageKey。
8. 运行题数、ID、答案、解析、批注、图片和分类校验。
9. 在本地 D1 验证迁移、题目查询和历史记录关联。
10. 提交并推送 GitHub。
11. 应用远程 D1 迁移。
12. 部署 Cloudflare Worker。
13. 在线抽查普通题、解析题、批注题、图片题、图表题和旧错题。
14. 保存本次导入的题数、ID 范围、迁移编号、提交号和异常清单。

## 十四、异常处理

- 题数不一致：停止导入，列出缺失、重复和无法映射的题号。
- 解析缺失：回到完整解析 Word 核对，不使用空字符串占位上线。
- 批注冲突：优先按内容相关度复核，无法确认则不绑定。
- 图片缺失：回到 Word/PDF 原页重新裁切，确认没有裁掉边界或图例。
- ID 冲突：重新扫描数据库和源文件，绝不覆盖其他题目。
- 本地校验失败：修复后重新从本地校验开始，不推送半成品。
- 线上异常：保留提交号、迁移号、接口响应和题目 ID，定位后继续修复并再次验证。

## 十五、完成标准

只有同时满足以下条件，才算一次题库导入完成：

- 所有题块均已切分并有明确归属；
- 题干、选项、答案、分类、解析和批注绑定正确；
- 格式和解析高亮能正常显示；
- 图表、图形和图片均可访问；
- 题目 ID 未破坏历史记录和错题本；
- 本地 D1 校验通过；
- GitHub 已推送；
- 远程 D1 已更新；
- Worker 已部署；
- 线上接口和前端抽查通过。
